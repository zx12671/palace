## 二、AI 产品与业务理解（AI PM 方向核心）

### Q4：如何定义一个 AI 产品的成功指标？

> 我会分三层来定义：
>
> **业务层**：对齐公司核心 KPI。比如在安踏做推荐系统，最终指标是 GMV 和转化率；在平安好医生做健康干预，最终指标是用户的治疗依从率。
>
> **产品层**：设定代理指标。因为业务指标往往受多个因素影响，产品层需要找到跟模型直接相关的代理指标——比如推荐的点击率、健康计划的 7 日完成率。
>
> **模型层**：技术指标。Recall@K、Precision、F1、延迟等。但这些指标只在产品指标方向正确时才有意义。
>
> 关键经验是：**不要把模型指标等同于产品成功**。我在安踏做过一版推荐模型，Recall@10 提升了 20%，但用户反馈「推荐的都差不多」。后来加了多样性惩罚项，Recall 降了 3%，但用户停留时长涨了 15%，最终 GMV 反而更高。这件事让我理解，技术指标必须通过 A/B 测试验证其对业务的真实影响。

---

### Q5：模型效果和用户体验冲突了怎么办？

> 原则是**用户体验优先，但要量化妥协成本**。
>
> 举个真实例子：在安踏做动态定价 AI 时，模型给出的最优价格策略会在某些时段产生剧烈价格波动，虽然理论上能最大化毛利，但用户看到价格变来变去就不信任了。
>
> 我的做法是：
>
> 1. 给模型加「价格波动率」上限约束，同一商品 24 小时内价格变动不超过 15%
> 2. 做 A/B 测试对比「无约束」vs「有约束」
> 3. 结果：有约束版本毛利略低 2%，但用户复购率提升 8%，30 天净收益反而更高
>
> **核心思路**：先量化冲突的两端各损失多少，再通过实验找平衡点，而不是靠直觉拍板。

---

### Q6：你怎么推动业务部门采纳 AI 方案？

> 最大障碍不是技术而是信任。业务部门不信「这个模型真的能用」。
>
> 我在安踏总结出一套「三步法」：
>
> **第一步：用他们的语言说话。** 不说「模型 AUC 达到 0.92」，而说「这个工具能让选品团队每周少开 3 天会，错误率降一半」。
>
> **第二步：先做小赢。** 不一上来就做全量推广，先挑一个小场景做 PoC，让他们看到效果。比如我在安踏先做了一个「清仓商品推荐」的小模块，两周上线，立刻帮业务清掉了一批滞销库存。
>
> **第三步：让业务参与定义成功指标。** 如果指标是你定的，模型效果好他们也不认。但如果指标是他们自己定的——比如「库存周转率提升 10%」——达标了他们就没法否认。
>
> 在安踏，用这个方法 6 个月内把 3 个核心业务线都接入了 AI 中台。

### Q7：如何从 0 到 1 搭建一个企业的 AI 能力？

> 我在安踏就做过这件事。到岗时 AI 团队只有 2 个人，走的时候是 10 人，建成了公司级 AI 中台。
>
> **路径是**：
>
> 1. **前 2 个月：选对第一个战场。** 不做大而全，而是找业务痛点最大、数据最现成、决策人最配合的场景。我选了「商品推荐」，因为数据现成（用户行为日志）、业务方急需（转化率下滑）、决策人支持（电商负责人愿意配合实验）。
> 2. **第 3-4 个月：快速出成果。** 用最简单的方案先跑通端到端。不追求最优模型，追求最快上线。第一版推荐模型其实就是 item-CF + 规则排序，但因为上线快、有 A/B 对照，业务方很快看到了转化率 +12% 的结果。
> 3. **第 5-8 个月：扩规模建中台。** 有了第一个成功案例，开始招人、建基础设施——特征平台、模型训练流水线、A/B 测试框架。同时横向扩展到定价、库存等场景。
> 4. **第 9-12 个月：沉淀方法论。** 把重复的模式抽象成可复用组件——比如标准化的数据接入流程、模型评估 checklist、业务对接 SOP。
>
> **关键心得**：不要一上来就建平台，先用一个成功案例证明 AI 的价值，再用中台来支撑规模化。

## 三、技术深度题（证明不脱离一线）

### Q8：介绍一下你的 RAG 系统是怎么设计的？

> 以 Palace 项目为例，我设计了一个两阶段检索 + 多智能体协作的架构：
>
> **检索阶段**：
>
> - 第一阶段：BM25 粗筛，从文档库中召回 top-50 候选
> - 第二阶段：用 Cross-Encoder（基于 bge-reranker）对 top-50 精排，取 top-5 传给 LLM
> - 为什么不直接用向量检索？因为在结构化文档（如政策条文、操作指南）场景下，BM25 的关键词匹配准确率更高，向量检索容易召回语义相似但事实无关的段落
>
> **生成阶段**：
>
> - Prompt 模板结构化：先放检索到的文档原文（带引用标记），再放用户问题，最后要求模型引用具体来源回答
> - 加入幻觉检测：对模型输出逐句检查是否有检索文档中的依据，无依据的标记为「待验证」
>
> **评估**：
>
> - 用黄金测试集（200 条人工标注的 QA 对）评估 F1，从基线 0.62 提升到 0.73
> - 幻觉率从 18% 降到 7%

---

### Q9：多智能体编排中，Agent 之间如何协作？

> 在 Palace 中我设计了一个流水线式的多智能体架构：
>
> **架构**：9 个 Agent 按角色串行协作——
>
> - 3 个「决策 Agent」：起草方案 → 审议反驳 → 综合定稿（对抗式协作）
> - 6 个「执行 Agent」：分别负责分工、资源、流程、排期、风控、工具（并行可选）
>
> **关键设计决策**：
>
> - **为什么用固定流水线而不是动态路由？** 因为决策流程是确定的，不确定的是每个环节的输出内容。确定性的流程 + 开放性的输出，比完全动态的 Agent 编排更可控、更可调试。
> - **为什么要对抗式设计？** 让「审议 Agent」专门找「起草 Agent」的漏洞。一个 Agent 同时做方案和审议，输出会趋于平庸。分离角色后，审议质量显著提升。
>
> **状态管理**：
>
> - 设计了 `DecisionSession` 状态机，支持暂停、恢复、回退
> - 用户可以在任何节点驳回上一步重来，或注入自己的意见
> - 每一步输出都持久化为 JSON，支持断点续跑
>
> **工程挑战**：
>
> - 9 次串行 LLM 调用，总耗时约 3 分钟。优化方案是六部 Agent 改为并行调用，预计提速 3-4 倍
> - LLM 输出格式不稳定，用 `unwrap()` + `deep_get()` 做自适应 JSON 解析

### Q10：向量数据库怎么选型？不同场景怎么选？

> 我的选型框架是按三个维度评估：
>
>
> | 维度    | Milvus  | Pinecone  | Chroma   | FAISS  |
> | ----- | ------- | --------- | -------- | ------ |
> | 数据规模  | 亿级      | 千万级       | 十万级      | 亿级（单机） |
> | 运维复杂度 | 高（需K8s） | 零（SaaS）   | 低（嵌入式）   | 低      |
> | 过滤能力  | 强（混合查询） | 中         | 弱        | 无      |
> | 适用场景  | 生产级大规模  | 快速原型/中小规模 | PoC/Demo | 研究/离线  |
>
>
> **我的实际选择逻辑**：
>
> - PoC 阶段用 Chroma，五分钟跑起来验证效果
> - 生产环境数据量 <1000 万条用 Pinecone（省运维），>1000 万条用 Milvus
> - 需要混合查询（向量 + 标签/时间/价格过滤）优先 Milvus
> - 纯离线实验用 FAISS，性能最好

### Q11：Prompt Engineering 你有什么实战经验？

> 三个核心技巧：
>
> **1. 结构化输出约束**：直接在 Prompt 里给 JSON Schema 示例。在 Palace 中，每个 Agent 的 Prompt 末尾都附带输出格式要求和示例，JSON 格式遵从率从 60% 提升到 95%。
>
> **2. 角色隔离**：一个 Prompt 只给一个角色。不要让一个 Agent 既出方案又审议。Palace 的门下省 Prompt 明确写了「你只负责找漏洞，不得定案」，输出的风险识别质量比「请你审议这个方案」高出很多。
>
> **3. 用 Few-shot 而不是长指令**：与其写 500 字描述期望输出的样子，不如直接给 2-3 个真实的输入-输出示例。在安踏做商品描述生成时，3-shot 的效果比纯指令好 30%。

### Q13：说一个跨部门协作推动 AI 落地的案例

> 在安踏做 AI 中台时，最难的不是技术，而是让三个互相不信任的部门配合。
>
> **背景**：要做一个「智能补货」系统。需要商品部提供销售预测需求，供应链部提供库存数据，IT 部提供系统接口。三个部门都觉得自己不需要 AI。
>
> **我做了三件事**：
>
> 1. **翻译需求**：分别跟三个部门单独聊，用他们的语言描述同一件事。跟商品部说「帮你减少手动算补货量的时间」，跟供应链说「减少因过量补货导致的仓库积压」，跟 IT 说「给你一个标准化数据接口，不用每次手动拉数据」。
> 2. **做一次联合 workshop**：把三个部门的人拉到一个屋子里，用一张流程图展示数据怎么流、AI 在哪个环节介入、各方的责任边界。30 分钟内达成共识。
> 3. **设立共同指标**：不给三个部门设各自的 AI 指标，而是设一个共同指标——「补货准确率提升 20%」。大家是同一条船上的人。
>
> **结果**：3 个月上线，补货准确率提升了 23%，库存周转率提升 15%。

### Q14：你如何评估一个 AI 项目是否值得做？

> 我用一个「四象限」评估框架：
>
>
> |           | 数据现成      | 数据需建设    |
> | --------- | --------- | -------- |
> | **业务价值高** | 立刻做       | 值得投入建设数据 |
> | **业务价值低** | 可以练手/demo | 不做       |
>
>
> 然后在「立刻做」象限里，按三个维度排优先级：
>
> 1. **业务紧迫度**：业务方是不是已经在为这个问题头疼？
> 2. **可衡量性**：成功指标能不能在 1 个月内看到数字变化？
> 3. **决策人配合度**：业务方负责人愿不愿意花时间跟你一起定义需求？
>
> 在安踏，我用这个框架砍掉了 60% 的「伪需求」，集中资源在 3 个真正有价值的场景上。

## 五、行业深度题（垂直领域壁垒）

### Q15：大健康/医疗 AI 的落地有什么特殊挑战？

> 三个核心挑战：
>
> 1. **数据合规**：医疗数据有严格的隐私保护要求（个人信息保护法、医疗数据分级管理）。在平安好医生时，所有模型训练数据都必须脱敏，模型不能存储任何可反推个体身份的中间结果。这意味着很多需要用户画像的推荐方案在医疗场景里行不通。
> 2. **专业性门槛**：医疗场景对「幻觉」零容忍。一个推荐系统推荐错了商品顶多用户不买，但一个健康干预系统给了错误建议可能危及健康。所以在做 RAG 系统时，必须引入「可引用性」约束——每条建议必须能追溯到循证医学来源。
> 3. **评估标准不同**：不是越准越好，而是越安全越好。在平安好医生，我们的 AI 筛查模型宁可多报假阳性（让医生多看几个片子），也不能漏报真阳性（漏诊）。这跟零售场景的优化逻辑完全相反。

---

### Q16：零售行业 AI 有哪些高价值落地场景？

> 我在安踏的实践中总结，零售 AI 落地按价值排序：
>
> 1. **智能推荐**（直接影响 GMV）：个性化商品推荐、搜索排序、关联推荐。这是投入产出比最高的场景，安踏做完推荐后线上转化率涨了 12%。
> 2. **动态定价**（提升毛利）：基于库存水平、竞品价格、用户敏感度自动调价。但必须加约束条件，不能让用户感觉价格混乱。
> 3. **需求预测 + 智能补货**（降低库存成本）：用历史销售 + 外部因素（天气、节假日、营销活动）预测各 SKU 各门店的需求量。安踏做完后库存周转率提升 15%。
> 4. **智能客服**（降低人力成本）：但只适合标准化问题，复杂售后仍需人工。
>
> **关键洞察**：零售 AI 的难点不在模型，在于数据链路——从 POS 机、ERP、WMS 到线上行为数据的打通。很多公司模型做得很好，但数据链路不通，模型跑不起来。

## 六、编码与系统设计题

### Q17：设计一个企业级 RAG 系统的架构

> **分层架构**：
>
> ```
> 用户查询
>     │
>     ▼
> ┌─────────────┐
> │  Query 改写  │  同义词扩展 + 意图识别
> └──────┬──────┘
>        │
>        ▼
> ┌─────────────┐     ┌─────────────┐
> │  BM25 粗召  │────→│  向量精排    │  Cross-Encoder
> └──────┬──────┘     └──────┬──────┘
>        │                    │
>        ▼                    ▼
> ┌──────────────────────────────────┐
> │          Prompt 组装              │
> │  Context: [检索文档]              │
> │  Question: [用户问题]             │
> │  Instruction: [引用来源回答]       │
> └──────────────┬───────────────────┘
>                │
>                ▼
> ┌─────────────┐     ┌─────────────┐
> │  LLM 生成   │────→│  幻觉检测    │  逐句验证是否有引用
> └──────┬──────┘     └──────┬──────┘
>        │                    │
>        ▼                    ▼
>    最终回答 + 引用来源
> ```
>
> **关键设计点**：
>
> - 混合检索（BM25 + 向量），不是二选一
> - 文档预处理：按段落切分 + 重叠窗口，保留上下文
> - 缓存层：高频查询缓存，降低 LLM 调用成本
> - 监控：记录每次查询的检索命中率、生成延迟、用户反馈

### Q18：手写一段代码：实现一个简单的 Agent 调度器

> （面试时可以用 Python 白板写）

```python
class Agent:
    def __init__(self, name, prompt_template):
        self.name = name
        self.prompt_template = prompt_template

    def run(self, input_data, llm_client):
        prompt = self.prompt_template.format(**input_data)
        return llm_client.generate(prompt)


class Pipeline:
    def __init__(self):
        self.steps = []

    def add(self, agent, input_mapper=None):
        self.steps.append((agent, input_mapper))
        return self

    def execute(self, initial_input, llm_client):
        context = {"input": initial_input}
        results = []
        for agent, mapper in self.steps:
            agent_input = mapper(context) if mapper else context
            result = agent.run(agent_input, llm_client)
            context[agent.name] = result
            results.append((agent.name, result))
        return context, results
```

> 核心思路：
>
> - Agent 是无状态函数：输入 → Prompt → LLM → 输出
> - Pipeline 维护全局 context，每步的输出都注入 context
> - `input_mapper` 让每个 Agent 可以从 context 中选择性地获取需要的数据
> - 扩展：加 `on_error` 回调、加 checkpoint 持久化、加并行执行
