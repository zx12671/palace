# 我用唐朝三省六部制，造了一个 AI 朝廷帮我做人生决策

> 200 行 Python，9 个 AI 智能体，跑出一份比我自己想 3 天还周全的决策报告。

---

要不要裸辞创业？租房续约还是搬家？孩子上公立还是民办？

这些问题有一个共同点：**选项不多，但每个选项背后的变量多到你根本想不完。**

你大概率会这样做——

打开一个文档，列个表，写写利弊，然后……盯着屏幕发呆，最后凭直觉选了一个。

我之前也是。直到有一天我在读《资治通鉴》的时候突然想到一件事：

**唐朝的皇帝，一个人治理一整个帝国，他是怎么做决策的？**

答案是——他不自己做决策。他有一整套制度帮他做。

## 三省六部：人类历史上最优雅的决策架构

唐朝的三省六部制，本质上是一个**多角色协作的决策流水线**：

```
中书省（起草）→ 门下省（审议）→ 尚书省（定稿）→ 六部（执行）
```

中书省负责拟方案，不做判断。
门下省负责挑毛病，专门唱反调。
尚书省负责拍板，综合正反意见做最终裁决。
六部负责落地，从人事、预算、流程、执行、风控到工具，六条线并行推进。

**这不就是一个天然的多智能体系统吗？**

每个「省」、每个「部」，就是一个有明确角色和职责的 Agent。输入结构化，输出结构化，上一个的输出就是下一个的输入。

于是我花了一个周末，用 200 行 Python + 阿里云千问大模型，把这套制度复刻成了一个 AI 系统。

我叫它 **Palace**。

## 它是怎么工作的？

你给系统一个「议题」，就像皇帝把一份奏折丢到朝堂上：

```json
{
  "title": "要不要裸辞出来做 AI 方向创业",
  "background": "28岁，年薪40万，存款30万，房贷月供6000...",
  "constraints": ["存款只撑10个月", "房贷不能断", "AI窗口期6-12个月"],
  "goals": ["判断是否合适", "最小化风险的启动方案"]
}
```

然后系统自动跑完整个流程：

```
==================================================
  议题: 要不要裸辞出来做 AI 方向创业
  模型: qwen3-max
==================================================

[三省流程]
  ▶ 中书省·草案 ...
  ✓ 中书省·草案 完成        ← 生成了3套方案
  ▶ 门下省·审议 ...
  ✓ 门下省·审议 完成        ← 指出了5条风险
  ▶ 尚书省·定稿 ...
  ✓ 尚书省·定稿 完成        ← 综合裁决，选出最优

[六部执行]
  ✓ 吏部·分工 完成          ← 谁做什么
  ✓ 户部·资源 完成          ← 要花多少钱
  ✓ 礼部·流程 完成          ← 怎么推进
  ✓ 兵部·执行 完成          ← 里程碑排期
  ✓ 刑部·风险 完成          ← 可能踩什么坑
  ✓ 工部·工具 完成          ← 用什么工具落地
```

**3 分钟后，你得到一份完整的决策报告。**

不是那种「建议你综合考虑」的废话，而是结构化的方案对比、风险清单、预算估算、执行排期——全套。

## 我拿它做了什么决策？

说实话，我自己最先当小白鼠。

### 场景一：创业技术选型

我就是用这个系统帮自己做了 Palace 本身的技术选型。

中书省给了我两套方案：FastAPI + HTMX 轻量路线 vs Streamlit 快速原型。

门下省直接指出：「两种方案都依赖轻量状态管理，并发时容易丢数据」「过度规避重型框架可能让未来扩展寸步难行」。

尚书省综合裁决：选 FastAPI 方案，但加上可插拔的状态管理抽象层。

六部接着拆解：户部估算预算 8000-25000 元、6-12 周；兵部列出了 5 个里程碑；刑部识别出 4 个关键风险……

**比我一个人闷头想两天靠谱多了。**

### 场景二：28 岁要不要裸辞创业

这是我一个朋友的真实困境。我让他把情况录入系统。

中书省给了三套方案：全职创业、请 3 个月无薪假试水、先加入一个 AI 创业公司。

门下省提出了一个我们都没想到的风险：「天使投资人承诺的 50 万没有 TS（条款清单），口头承诺不算数。」

**就这一条，值了。**

### 场景三：深圳租房

续租老房子还是搬到公司附近？听起来简单，但六部一拆，你才发现有多少维度你没考虑——

户部算了两个方案的 12 个月总成本（含通勤、外卖差价）；礼部列了搬家的完整流程清单；兵部给了一个「先签约再搬家」的无缝切换时间线；刑部提醒要查新房合同里的涨租条款……

## 为什么不直接问 ChatGPT？

你当然可以直接问 ChatGPT「我该不该裸辞」。

它会给你一个看起来很全面的回答。但问题在于：

**一个 Agent 扮演所有角色 = 没有角色。**

它既要帮你想方案，又要帮你挑毛病，还要帮你做判断。这就像让一个人同时当原告、被告和法官——最后的输出一定是面面俱到、但毫无锋芒的中庸之辞。

Palace 的核心设计哲学是：**把「想方案」和「挑毛病」交给不同的 Agent。**

中书省只管头脑风暴，放飞自我。
门下省只管找茬反驳，不留情面。
尚书省综合双方意见做最终裁决。

**这种对抗性的协作，才能产出真正有质量的决策。**

就像唐太宗和魏征的关系——皇帝不需要一个什么都说好的大臣，需要一个敢说「陛下此举不妥」的谏臣。

## 交互模式：你就是皇帝

最新版本加入了交互模式。系统不再是黑盒跑完就给结果，而是在每个关键节点暂停，等你拍板：

```
中书省给出3套草案
  → 你选择倾向哪套，补充意见
门下省审议完毕
  → 你决定是否驳回，让中书省重来
尚书省定稿
  → 你确认最终方案，或打回重审
六部各出执行计划
  → 你逐一审批，不满意可以让某个部门重做
```

随时 `Ctrl+C` 中断，系统自动保存断点，下次从中断处继续。

**你就是坐在龙椅上的那个人。九个 AI 大臣给你干活，但最终拍板的永远是你。**

## 技术极简主义：零框架、零依赖

整个系统没有用 LangChain，没有用 CrewAI，没有任何第三方依赖。

- **语言**：Python 3.9+，标准库 `urllib` 发请求
- **模型**：阿里云 qwen3-max（262K 上下文，千问旗舰）
- **架构**：每个 Agent 就是一个 Markdown 模板 + 一次 LLM 调用
- **状态机**：自己写的 `DecisionSession`，支持暂停、恢复、回退

```
palace/
├── palace/
│   ├── llm.py           # 50行，调LLM
│   ├── agents.py        # 40行，跑Agent
│   ├── session.py       # 状态机，支持交互
│   └── renderer.py      # 生成Markdown报告
├── templates/
│   └── agent_prompts/   # 9个Agent的提示词模板
├── interactive.py       # 交互式CLI
└── mvp.py              # 批处理模式
```

为什么不用 LangChain？因为这个场景的 Agent 协作模式是确定的——线性流水线，不需要动态路由、工具调用、记忆检索那些复杂机制。**用 200 行代码能解决的事，不需要引入一个 30000 行的框架。**

Agent 框架的价值在于处理不确定性。而三省六部的妙处恰恰在于——**流程是确定的，但每个环节的输出是开放的。**

## 一些有趣的发现

用了一段时间，我发现几个出乎意料的事：

**1. AI 的「对抗」比「协作」更有价值**

门下省经常能指出中书省方案里我自己完全忽略的盲区。不是因为 AI 更聪明，而是因为它被赋予了「找茬」这个单一目标，就会拼命去找——而你自己思考时，潜意识会避开那些让你不舒服的风险。

**2. 六部的价值在于「强制覆盖」**

你自己做决策时，想到预算就不想想风险，想到风险就忘了流程。六部强制你把每个维度都过一遍，不允许遗漏。

**3. 交互模式比自动模式有用 10 倍**

一开始我做的是全自动批处理。跑完 3 分钟得到一份报告，看完就忘了。后来加了交互模式，每一步都要你参与——你会发现自己在审阅门下省意见时，脑子里冒出了很多之前从来没想到的东西。

**系统的价值不在于给你答案，在于逼你把问题想清楚。**

## 开源 & 下一步

Palace 已经开源，MIT 协议，代码在 GitHub 上。

你只需要：

```bash
# 设置API Key（阿里云百炼免费送100万token）
export DASHSCOPE_API_KEY="你的key"

# 跑一个决策试试
python3 interactive.py --issue tests/fixtures/issue_quit_startup.json
```

下一步计划：

- **Web 界面**：不用命令行，浏览器里直接用
- **六部并行**：现在是串行调用 9 次 LLM，并行后速度提升 3-4 倍
- **自定义部门**：你可以增加自己的「部门」，比如加一个「太医院」专门评估健康风险
- **多模型混用**：三省用旗舰模型保证质量，六部用轻量模型降低成本
