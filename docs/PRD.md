# Palace — 多智能体个人决策系统 PRD

> 产品需求文档 v1.0 | 2026-02-24

---

## 1. 产品概述

### 1.1 一句话定义

Palace 是一个模拟唐朝「三省六部」制度的多智能体个人决策辅助系统——用户输入一个待决策事项，系统通过 9 个 AI Agent 协作完成从方案生成到执行计划的全流程，输出一份结构化的决策报告。

### 1.2 产品愿景

让每个人都拥有自己的「朝廷智囊团」。面对人生中的重大决策（职业选择、技术选型、租房搬家、孩子择校……），不再凭直觉拍脑袋，而是通过多角色、多视角的 AI 协作，获得经过草案→审议→定稿→执行规划的完整决策方案。  
面对项目的具体实践（项目计划、技术选型、 、孩子择校……），不再凭直觉拍脑袋，而是通过多角色、多视角的 AI 协作，获得经过草案→审议→定稿→执行规划的完整决策方案。

### 1.3 核心隐喻


| 唐朝制度 | 系统角色 | 职责                    |
| ---- | ---- | --------------------- |
| 三省   | 决策层  | 方案生成→审议反驳→最终裁决        |
| 六部   | 执行层  | 分工→资源→流程→执行→风险→工具     |
| 皇帝   | 用户   | 提出议题（Issue），查阅朝廷的决策结果 |

## 3. 系统架构

### 3.1 多智能体流水线

```
用户输入 Issue
       │
       ▼
┌─────────────┐
│  中书省·草案  │  生成 ≥2 套可执行方案（含假设、利弊、粗步骤）
└──────┬──────┘
       │ Draft
       ▼
┌─────────────┐
│  门下省·审议  │  审议草案，指出 ≥3 条风险，提出 ≥2 条修订建议
└──────┬──────┘
       │ Review
       ▼
┌─────────────┐
│  尚书省·定稿  │  整合草案+审议 → 圣裁优先方案 + 备选方案
└──────┬──────┘
       │ Final
       ▼
┌──────────────────────────────────────────┐
│              六 部 执 行                    │
│                                          │
│  吏部·分工    →  角色/任务分配清单          │
│  户部·资源    →  预算、时间、资源缺口估算    │
│  礼部·流程    →  流程步骤与沟通要点         │
│  兵部·执行    →  有序执行步骤与里程碑        │
│  刑部·风险    →  风险识别与规避措施          │
│  工部·工具    →  可落地的工具/模板/交付物     │
└──────────────────────────────────────────┘
       │
       ▼
  决策报告（Markdown + JSON）
```

### 3.2 数据流

```
Issue (JSON)
  → 中书省 → Draft (JSON)
  → 门下省 → Review (JSON)
  → 尚书省(Draft + Review) → Final (JSON)
  → 六部(Final) → ExecPlan (JSON)
  → 汇总渲染 → decision.md
```

每个 Agent 的输入输出都是结构化 JSON，通过 Prompt 模板驱动 LLM 生成。

### 3.3 Agent 定义

每个 Agent 由一个 Markdown 模板文件定义，包含：


| 字段      | 说明                      |
| ------- | ----------------------- |
| 角色      | Agent 在系统中的身份           |
| 目标      | 需要完成的具体任务               |
| 输入 JSON | 接收的数据结构                 |
| 输出 JSON | 产出的数据结构                 |
| 约束      | 输出质量的硬性要求               |
| 提示词     | 发送给 LLM 的 system prompt |


模板存放于 `templates/agent_prompts/`，可独立编辑和扩展。

## 4. 数据模型

### 4.1 Issue（议题）

用户输入的待决策事项，是整个流程的起点。

```json
{
  "id": "career-001",
  "domain": "career",
  "title": "要不要裸辞出来做 AI 方向创业",
  "background": "详细背景描述...",
  "constraints": ["约束条件1", "约束条件2"],
  "goals": ["目标1", "目标2"],
  "inputs": [
    {"type": "context", "name": "利好因素", "summary": "..."},
    {"type": "context", "name": "风险因素", "summary": "..."}
  ],
  "deadline": "2周内决定",
  "priority": "critical"
}
```

### 4.2 Draft（中书省·草案）

```json
{
  "drafts": [
    {
      "name": "方案名称",
      "summary": "方案概述",
      "pros": ["优势1", "优势2"],
      "cons": ["劣势1", "劣势2"],
      "rough_steps": ["步骤1", "步骤2"]
    }
  ]
}
```

### 4.3 Review（门下省·审议）

```json
{
  "risks": ["风险1", "风险2", "风险3"],
  "revision_suggestions": ["修订建议1", "修订建议2"]
}
```

### 4.4 Final（尚书省·定稿）

```json
{
  "imperial_choice": {
    "summary": "圣裁优先方案描述",
    "reasoning": "选择理由"
  },
  "alternative": {
    "summary": "备选方案描述",
    "reasoning": "备选理由"
  }
}
```

### 4.5 ExecPlan（六部执行计划）

```json
{
  "issue_id": "career-001",
  "liubu": {
    "libu":        {"roles": [...]},
    "hubu":        {"budget": "", "time": "", "gaps": [...]},
    "libu_ritual": {"process": [...], "comms": [...]},
    "bingbu":      {"steps": [...], "milestones": [...]},
    "xingbu":      {"risks": [...], "mitigations": [...]},
    "gongbu":      {"tools": [...], "templates": [...]}
  }
}
```

## 5. 输出物

每次决策运行产生一个独立目录，包含：


| 文件                                | 内容          |
| --------------------------------- | ----------- |
| `01_zhongshu_draft.json`          | 中书省草案原始输出   |
| `02_menxia_review.json`           | 门下省审议原始输出   |
| `03_shangshu_final.json`          | 尚书省定稿原始输出   |
| `04_libu.json` ~ `09_gongbu.json` | 六部各部门原始输出   |
| `exec_plan.json`                  | 六部执行计划汇总    |
| `decision.md`                     | 人类可读的完整决策报告 |


决策报告 (`decision.md`) 包含：背景 → 草案方案详情 → 审议意见 → 圣裁定稿 → 六部各部门完整计划。

## 6. 技术实现（当前 MVP）

### 6.1 技术栈


| 组件      | 选型                         |
| ------- | -------------------------- |
| 语言      | Python 3.9+                |

## 7. 产品演进路线

### Phase 1: CLI MVP（当前 ✅ 已完成）

- 9 个 Agent 流水线跑通
- 结构化 JSON 输入输出
- Markdown 决策报告生成
- 各部门独立文件保存 + 进度日志
- 5 个真实场景 fixture 验证通过

### Phase 2: Web 界面

- Flask/FastAPI 后端，提供 REST API
- 简洁的 Web 前端（输入 Issue → 实时进度 → 查看报告）
- 会话隔离，支持多用户同时使用
- 历史决策记录与回溯
- 手机端适配

### Phase 3: 交互增强

- 用户可在审议阶段插入个人意见（参与门下省讨论）
- 支持对单个部门结果进行「驳回重来」
- 支持追加补充信息后重新决策
- 决策对比：同一议题不同参数下的方案对比

### Phase 4: 智能增强

- Agent 可访问外部工具（搜索引擎、计算器、数据库）
- 支持上传文档作为决策依据（PDF、图片等）
- 多模型混用（不同 Agent 用不同模型，成本优化）
- 六部并行执行（当前为串行，可提速 3-4 倍）

### Phase 5: 生态扩展

- 自定义 Agent（用户可添加自己的「部门」）
- Agent 模板市场（社区共享不同决策框架）
- 团队协作（多人参与同一决策流程）
- API 开放（接入其他应用的决策场景）

## 8. 关键指标

### 8.1 MVP 验收标准


| 指标      | 目标                      |
| ------- | ----------------------- |
| 端到端成功率  | 5 个场景全部跑通，无报错           |
| 单次运行时间  | < 5 分钟                  |
| 草案方案数   | ≥ 2 套                   |
| 审议风险数   | ≥ 3 条                   |
| 圣裁方案完整性 | 必须包含理由说明                |

| 输出可读性   | Markdown 报告结构完整，人类可直接阅读 |


### 8.2 未来关注指标


| 指标    | 说明                          |
| ----- | --------------------------- |
| 决策采纳率 | 用户实际采纳系统建议的比例               |
| 重复使用率 | 同一用户是否多次使用                  |
| 方案差异度 | 不同 Agent 生成的内容是否有实质差异（而非重复） |
| 用户满意度 | 决策报告是否对用户有实际帮助              |

## 9. 风险与约束


| 风险          | 影响                       | 缓解措施                                                |
| ----------- | ------------------------ | --------------------------------------------------- |
| LLM 输出结构不稳定 | Agent 返回的 JSON 字段名/结构不固定 | `unwrap()` + `deep_get()` 自适应解析；后续可加 JSON Schema 校验 |
| API 调用成本    | 9 次 LLM 调用/次决策           | 控制 max_tokens；未来可对低优先级部门使用更便宜的模型                    |
| 串行调用耗时长     | 单次运行 ~3 分钟               | Phase 4 引入六部并行                                      |
| 模型幻觉        | Agent 可能生成不切实际的建议        | 门下省审议机制本身就是一层校验；用户最终拍板                              |
| 单一 API 依赖   | DashScope 不可用时系统失效       | 预留多 API 后端切换能力（已有 OpenAI 兼容接口）                      |

## 10. 术语表


| 术语       | 说明                                        |
| -------- | ----------------------------------------- |
| Issue    | 用户提交的待决策事项                                |
| Draft    | 中书省生成的草案（含多套方案）                           |
| Review   | 门下省的审议意见（风险 + 修订建议）                       |
| Final    | 尚书省的最终定稿（圣裁方案 + 备选）                       |
| ExecPlan | 六部生成的执行计划汇总                               |
| 圣裁       | 系统综合草案与审议后做出的最终推荐                         |
| 三省       | 中书省（草案）、门下省（审议）、尚书省（定稿）                   |
| 六部       | 吏部（分工）、户部（资源）、礼部（流程）、兵部（执行）、刑部（风险）、工部（工具） |
| Agent    | 一个由 Prompt 模板驱动的 LLM 调用单元，扮演特定角色          |
