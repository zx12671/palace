# Palace Web 化项目计划

> 基于三省六部决策结论制定 | 圣裁方案：FastAPI + HTMX + 原生 Python Agent
>
> 约束：1 人开发，每天 ≤3h，2 周出 MVP，月成本 ≤500 元

---

## 总览


| 项目    | 内容                               |
| ----- | -------------------------------- |
| 目标    | 从 CLI MVP 进化为可给朋友使用的 Web 产品      |
| 技术栈   | FastAPI + HTMX + Jinja2 + SQLite |
| 部署    | 阿里云 FC + OSS（主），ECS（备）           |
| CI/CD | GitHub Actions 自动构建部署            |
| 总预算   | ¥8,000–¥15,000（控制在下限附近）          |
| 总工期   | 6 周（Phase 1–3），后续持续迭代            |


---

## Phase 1：基础架构（第 1–2 周）

> 对应兵部 Step 1–2，礼部流程 1–3

### 目标

搭建 FastAPI 骨架，将现有 `palace/` 核心逻辑接入 Web 后端，实现可插拔状态管理。

### 任务清单


| #   | 任务                       | 工时估算 | 产出物                                        | 风险缓解              |
| --- | ------------------------ | ---- | ------------------------------------------ | ----------------- |
| 1.1 | 创建 FastAPI 项目结构          | 2h   | `web/main.py`, `web/routes/`, `web/state/` | -                 |
| 1.2 | 设计 StateInterface 抽象基类   | 3h   | `web/state/interface.py`                   | R001: 统一读写接口      |
| 1.3 | 实现 SQLiteStateStore      | 4h   | `web/state/sqlite_store.py`                | R001: 可插拔适配器      |
| 1.4 | 封装 palace 核心为 FastAPI 路由 | 4h   | `web/routes/decision.py`                   | -                 |
| 1.5 | SSE/流式进度推送接口             | 3h   | `web/routes/stream.py`                     | Agent 运行耗时长，需实时反馈 |
| 1.6 | 接入现有 DecisionSession     | 3h   | 复用 `palace/session.py`                     | R002: 复用已有状态机     |
| 1.7 | 单元测试：状态管理 + API 路由       | 3h   | `tests/test_web.py`                        | -                 |


### 里程碑 M1：项目骨架与状态抽象层完成

- **验收标准**：`pytest` 通过，API 可接收 Issue JSON 并返回 Agent 运行结果
- **预计完成**：第 2 周末

---

## Phase 2：前端交互（第 3–4 周）

> 对应兵部 Step 3–4，礼部流程 3–4

### 目标

用 HTMX + Jinja2 构建极简 Web 界面，实现完整的交互式决策流程。

### 任务清单


| #    | 任务                | 工时估算 | 产出物                           | 备注                  |
| ---- | ----------------- | ---- | ----------------------------- | ------------------- |
| 2.1  | 主页模板：议题输入表单       | 2h   | `web/templates/index.html`    | HTMX 表单提交           |
| 2.2  | 决策流程页：分步展示 + 实时进度 | 5h   | `web/templates/decision.html` | SSE 推送 Agent 状态     |
| 2.3  | 交互节点：草案选择 UI      | 3h   | HTMX 局部更新片段                   | 卡片式方案对比             |
| 2.4  | 交互节点：审议意见 + 用户补充  | 3h   | 评论框 + 驳回按钮                    | 对应 `review_opinion` |
| 2.5  | 交互节点：圣裁定稿确认       | 2h   | 确认/驳回操作                       | 对应 `confirm_final`  |
| 2.6  | 交互节点：六部逐一审批       | 3h   | 折叠面板 + 审批按钮                   | 对应 `confirm_dept`   |
| 2.7  | 决策报告页：Markdown 渲染 | 2h   | `web/templates/report.html`   | 复用 `renderer.py`    |
| 2.8  | 历史记录页             | 2h   | `web/templates/history.html`  | 列出历史决策              |
| 2.9  | 基础 CSS 样式         | 3h   | `web/static/style.css`        | 简洁现代风格              |
| 2.10 | 前后端联调测试           | 3h   | 手动验收                          | 跑通完整流程              |


### 里程碑 M2 + M3

- **M2 验收**（第 3 周末）：三省流程 Web 端可交互
- **M3 验收**（第 4 周末）：含六部的完整流程在浏览器中跑通
- **预计完成**：第 4 周末

## 每周节奏（礼部流程规范）


| 节点       | 动作                              |
| -------- | ------------------------------- |
| 每周一      | 确认本周任务，更新进度                     |
| 每周五      | 检查里程碑达成情况，决定是否调整                |
| Phase 切换 | 回顾上一 Phase，确认验收标准通过后再进入下一阶段     |
| 范围变更     | 任何新增需求须评估影响，不得超出 MVP 边界（礼部流程 3） |


### MVP 范围红线（礼部 + 刑部联合划定）

- ✅ 单轮三省六部完整流程
- ✅ 单用户交互式决策
- ✅ 历史记录查看
- ✅ 一键部署
- ❌ 多用户并发（Phase 2 以后）
- ❌ 多轮驳回迭代（已有 CLI 交互支持，Web 端 Phase 2）
- ❌ 多模型混用
- ❌ 自定义 Agent/部门
